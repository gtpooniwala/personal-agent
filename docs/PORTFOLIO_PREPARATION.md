# Portfolio Preparation Guide

This is optional packaging guidance if the repository is presented externally. It is not a product roadmap and it intentionally avoids stale metrics or unsupported claims.

## What Already Presents Well
- Clear architecture and execution docs
- Real async runtime design with worker-pool responsiveness work
- Full-stack scope: FastAPI, LangGraph, PostgreSQL, Next.js
- Concrete deployment and trigger planning docs
- Worktree/PR workflow and validation discipline

## Safe Claims To Emphasize
- durable async run submission and polling
- tool-using personal-agent architecture
- document workflows and memory features
- practical engineering around runtime hardening, evals, and deployment planning

## Useful Improvements

### Demo Assets
- capture current screenshots of chat, document flow, and runtime progress
- add a short demo GIF only if it reflects current UI and behavior

### README Polish
- keep the top-level README concise and technically honest
- prefer architecture and workflow clarity over marketing language
- link to the strongest docs rather than duplicating them

### External Presentation
- add a short repository description and topics
- document what is implemented today versus what is planned
- avoid hardcoded coverage, latency, uptime, or "production-ready" claims unless they are measured and maintained

## Checklist Before Sharing Publicly
- confirm no credentials or private auth artifacts are in the repo
- verify setup instructions still work on a clean machine
- verify screenshots or demo assets match the current frontend
- verify docs reference the current runtime and roadmap
